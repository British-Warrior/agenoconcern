**UX-09: Error message sanitisation**

Audit error display across components. The `ApiResponseError` class already sets `message` from `data.error` which is typically a human-readable string from the server. The risk is when `error.message` contains a UUID or technical detail from the API response.

1. **CircleFormationModal.tsx** (line 80-81): Currently displays `createCircle.error.message` which could contain a UUID if the API returns one. Wrap with a sanitisation guard:
   ```typescript
   const UUID_PATTERN = /[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}/i;
   ```
   Add this pattern at the top of the file (outside the component). Then change the error display to:
   ```typescript
   {createCircle.error instanceof Error && !UUID_PATTERN.test(createCircle.error.message)
     ? createCircle.error.message
     : "Failed to form circle. Please try again."}
   ```

2. **CircleWorkspace.tsx** (line 49-65): The error display already has good fallback messages for 403 and generic errors. The `error.message` check for "403" is fine since ApiResponseError sets message from the API's error string. However, add a UUID sanitisation guard to the generic error path:
   ```typescript
   // In the non-403 branch, replace the hardcoded "Failed to load workspace." with:
   const rawMessage = error instanceof Error ? error.message : "";
   const safeMessage = rawMessage && !UUID_PATTERN.test(rawMessage)
     ? rawMessage
     : "Failed to load workspace.";
   ```
   Add the UUID_PATTERN constant at the top of the file.

3. **InterestButton.tsx** (lines 77-78, 95-96): Already uses `err.message` which comes from ApiResponseError. Add UUID guard:
   ```typescript
   } else if (err instanceof Error) {
     const msg = UUID_PATTERN.test(err.message)
       ? "Something went wrong. Please try again."
       : err.message;
     setErrorMessage(msg);
   }
   ```
   Add UUID_PATTERN at the top of the file. Apply to both the handleExpress and handleWithdraw catch blocks.

**UX-11: Circle formation member names verification**

After inspecting the code, `CircleFormationModal` receives `memberNames` from `TeamCompositionCard` which maps `composition.contributors.map((c) => c.name)`. The `TeamComposition.contributors` type has `name: string`. The names come from the API response for team compositions.

Add a defensive guard in `CircleFormationModal.tsx` -- if any `memberNames` entry is empty or looks like a UUID, display "Unknown contributor" instead:
```typescript
// Inside the member list rendering:
{memberNames.map((name, i) => {
  const displayName = (!name || UUID_PATTERN.test(name)) ? "Unknown contributor" : name;
  return (
    <li key={memberIds[i]} ...>
      ...
      <span className="text-xs font-semibold text-primary-700">
        {displayName.charAt(0).toUpperCase()}
      </span>
      ...
      <span className="text-sm text-neutral-800">{displayName}</span>
    </li>
  );
})}
```

This is a defensive fix. If the API returns correct names (likely), no change is visible. If it returns UUIDs or empty strings, the UI degrades gracefully.
