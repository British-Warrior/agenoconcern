<task type="auto">
  <name>Task 1: Fix PortalDashboard heading hierarchy</name>
  <files>packages/web/src/pages/portal/PortalDashboard.tsx</files>
  <action>
The PortalDashboard currently renders its h1 inside a `<header>` element ABOVE `<main>`. This means:
1. RouteChangeSync (`document.querySelector('main h1')`) cannot find the h1
2. The first headings inside `<main>` are h2 elements (Impact Report, Attention Flags) with no preceding h1

Fix: Restructure the component so the h1 is inside `<main>` as the first heading element. The institution name heading must be the first child of `<main>` (or wrapped in a section that is the first child). Keep the visual header bar with its dark background — move the h1 text into `<main>` and replace the header's h1 with a non-heading element (e.g. `<span>` or `<p>`) or use a visually-hidden h1 inside main plus aria-hidden on the header heading.

Recommended approach — simplest and cleanest:
- Inside `<main>`, add the institution name as an `<h1>` at the very top (before the stats grid). Use `className="text-2xl font-bold text-neutral-900 mb-2"` for consistent styling with other dashboard pages.
- In the `<header>`, change the `<h1>` to a `<p>` with the same classes (it is still visually present in the header bar but is no longer a heading). The semantic heading is now inside main.

Do NOT:
- Add a second h1 (one in header and one in main) — that violates the single-h1 rule
- Use aria-hidden on the main h1 — RouteChangeSync needs it visible to the accessibility tree
  </action>

4. The component still renders and the visual layout is preserved

Run: `cd packages/web && npx tsc --noEmit` — zero type errors
