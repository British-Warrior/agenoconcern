1. Create `packages/server/src/services/report-delivery.job.ts`:
   - Export `startReportDeliveryJob()` — registers `cron.schedule("0 * * * *", ...)` (hourly). Mirror the pattern in `wellbeing-reminder.job.ts`.
   - `runReportDeliveryJob()`:
     a. Acquire advisory lock with key 7171 via `db.execute(sql\`SELECT pg_try_advisory_lock(7171)\`)`. If not acquired, log and return. Always release in `finally` block.
     b. Query institutions WHERE reportDeliveryEnabled = true AND reportNextRunAt <= now() AND contactEmail IS NOT NULL.
     c. For each institution:
        - Compute date range based on cadence: weekly = previous Mon 00:00 UTC to Sun 23:59:59 UTC; monthly = 1st of previous month to last day of previous month.
        - Gather report data using the same logic as the existing GET /institutions/:slug/report.pdf route (contributor assignments, challenge count, hours, wellbeing band). Extract this into a helper function `gatherReportData(db, institutionId, startDate, endDate)` in the same file (or inline if simpler — use judgment).
        - Call `buildInstitutionReport(reportData)` and capture as Buffer using `pdfDocToBuffer()` helper (define in this file: collect stream chunks, resolve on 'end').
        - Send email via Resend with the PDF buffer as attachment. Use `getEnv().RESEND_API_KEY`. If key is not set, log warning and skip (graceful degradation, matching notification.service.ts pattern).
        - On success: insert a 'sent' row into reportDeliveryLogs, update institution's reportNextRunAt to next period (next Monday 08:00 UTC for weekly, 1st of next month 08:00 UTC for monthly).
        - On failure: insert a 'failed' row into reportDeliveryLogs with errorMessage and attemptNumber=1, set nextRetryAt using exponential backoff formula (15min base * 2^0 + jitter). (Retry processing is in Plan 17-02.)
     d. Also query reportDeliveryLogs WHERE status = 'failed' AND nextRetryAt <= now() AND nextRetryAt IS NOT NULL — process retries. For each retry row: re-gather data, re-generate PDF, re-send email. On success: insert new 'sent' log row, clear the failed row's nextRetryAt (set to null), update institution nextRunAt. On failure: if attemptNumber < 5, insert new 'failed' row with attemptNumber+1 and new nextRetryAt; if attemptNumber >= 5, insert 'failed' row with nextRetryAt = null (exhausted).

   - Helper: `pdfDocToBuffer(doc: PDFKit.PDFDocument): Promise<Buffer>` — collect chunks on 'data', resolve Buffer.concat on 'end', reject on 'error'. Do NOT call doc.end() inside this helper (the caller or buildInstitutionReport handles it — check if buildInstitutionReport calls doc.end(); if not, call it after piping).

   - Helper: `computeNextRetryAt(attemptNumber: number): Date | null` — MAX_ATTEMPTS=5, BASE_MS=15*60*1000, MAX_DELAY_MS=4*60*60*1000, jitter=Math.floor(Math.random()*60000), delay=Math.min(BASE_MS * 2^(attempt-1) + jitter, MAX_DELAY_MS). Return null if attemptNumber >= MAX_ATTEMPTS.

   - Helper: `computeNextRunAt(cadence: 'weekly' | 'monthly'): Date` — weekly: next Monday at 08:00 UTC; monthly: 1st of next month at 08:00 UTC.

   - Helper: `computePeriodRange(cadence: 'weekly' | 'monthly'): { startDate: Date; endDate: Date }` — weekly: previous Mon 00:00 UTC to previous Sun 23:59:59.999 UTC; monthly: 1st of previous month 00:00 UTC to last day of previous month 23:59:59.999 UTC.

2. Register in `packages/server/src/index.ts`:

   - Call `startReportDeliveryJob()` inside the `app.listen` callback, right after `startWellbeingReminderJob()`.

IMPORTANT: Check whether `buildInstitutionReport` calls `doc.end()` internally. Looking at the existing route, `doc.end()` is called AFTER `doc.pipe(res)` — so `buildInstitutionReport` does NOT call doc.end(). Therefore, in `pdfDocToBuffer`, you MUST call `doc.end()` after setting up the stream listeners.
