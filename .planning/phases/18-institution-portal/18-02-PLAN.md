  <action>
    Create `packages/server/src/routes/portal/portal-dashboard.ts` — a Router with `portalAuthMiddleware` applied to all routes.

    1. **`GET /dashboard`** — Returns `{ institutionName, stats: { contributors, challenges, hours } }` for `req.portalUser.institutionId`. Reuse the same query logic from `GET /admin/institutions` in admin.ts:
       - Look up institution name from `institutions` table
       - Get contributor IDs from `contributorInstitutions` where `institutionId` matches
       - Count unique challenges from `challengeInterests` for those contributors
       - Sum hours from `contributorHours` for those contributors
       - Return `{ institutionName, stats: { contributors: count, challenges: uniqueCount, hours: totalHours } }`
       - If no contributors assigned, return stats with all zeros (don't 422 like the PDF endpoint does for admin)

    2. **`GET /report.pdf`** — Downloads the institution's PDF impact report. No slug parameter needed — scope by `req.portalUser.institutionId`.
       - Look up institution by ID (get name, city, slug for filename)
       - Get contributor assignments, challenge interests, hours, wellbeing data — SAME query logic as `GET /admin/institutions/:slug/report.pdf` in admin.ts (extract the data-gathering portion)
       - If no contributors assigned, return 422 with helpful message
       - Call `buildInstitutionReport(reportData)` and pipe to response
       - Set Content-Type: application/pdf, Content-Disposition with institution slug in filename, Cache-Control: no-store

    3. **`GET /attention`** — Returns active (unresolved) attention flags for the institution. Query `ithinkAttentionFlags` where `institutionId = req.portalUser.institutionId` AND `clearedAt IS NULL`, joined with `contributors` for name. Order by `createdAt` DESC. Return same shape as admin `/attention` endpoint: `{ id, contributorId, contributorName, signalType, cohortSize, flaggedCount, createdAt }`. NO resolve endpoint — read-only.

    Mount in `portal-auth.ts`: Import `portalDashboardRoutes` and mount on the same portal router. The portal-auth.ts file should re-export a combined router. Alternatively, mount `portalDashboardRoutes` separately in express-app.ts under the same `/api/portal` prefix — choose whichever is cleaner. The simplest approach: in express-app.ts, mount both routers at `/api/portal`:
    ```typescript
    app.use("/api/portal", portalAuthRoutes);
    app.use("/api/portal", portalDashboardRoutes);
    ```
    This works because Express merges route handlers for the same prefix.

    - All three endpoints return 401 without portal cookie
    - All three endpoints return 401 with a contributor cookie (type discriminant)
