    1. **Shared types** (`packages/shared/src/types/challenge.ts`):
       - `Challenge` type matching the DB columns: id, createdBy, title, description, brief, domain, skillsNeeded (string[]), type ('paid'|'free'), deadline (string|null), circleSize, status ('draft'|'open'|'closed'|'archived'), interestCount, createdAt, updatedAt
       - `ChallengeInterest` type: id, challengeId, contributorId, status ('active'|'withdrawn'), note (string|null), matchScore (number|null), lastWithdrawnAt (string|null), createdAt, updatedAt
       - `ChallengeFilters` type: domain?, type?, page?, limit?
       - `TeamComposition` type: contributors array (id, name, skills, score), diversityScore, coverageScore, balanceScore
       - `ChallengeInterestResponse` type for the interest toggle: status, activeInterestCount?, maxCircles?, cooldownRemainingHours?
       - `ChallengeFeedResponse`: challenges array, page number, hasMore boolean

    2. **Shared Zod schemas** (`packages/shared/src/schemas/challenge.ts`):
       - `createChallengeSchema`: title (5-200 chars), description (20-5000 chars), brief (10-500 chars), domain (enum from DOMAIN_TAXONOMY), skillsNeeded (string[] max 20 items), type ('paid'|'free'), deadline (optional ISO date string), circleSize (int 2-10, default 4)
       - `updateChallengeSchema`: same fields but all optional (partial)
       - `interestNoteSchema`: note (optional string, max 500 chars)

    3. **Export** new types and schemas from `packages/shared/src/index.ts`.

    4. **Database schema** (`packages/server/src/db/schema.ts`):
       - Add `challengeTypeEnum` pgEnum: ['paid', 'free']
       - Add `challengeStatusEnum` pgEnum: ['draft', 'open', 'closed', 'archived']
       - Add `challengeInterestStatusEnum` pgEnum: ['active', 'withdrawn']
       - Add `challenges` table per RESEARCH Pattern 3 — use `jsonb` for skillsNeeded (consistent with existing profile pattern). Columns: id (uuid pk), createdBy (FK to contributors), title, description, brief, domain, skillsNeeded (jsonb string[]), type, deadline (date mode:date), circleSize (smallint default 4), status, interestCount (integer default 0), createdAt, updatedAt
       - Add `challengeInterests` table: id (uuid pk), challengeId (FK cascade), contributorId (FK cascade), status, note (text nullable), matchScore (smallint nullable), lastWithdrawnAt (timestamp nullable), createdAt, updatedAt
       - Add unique constraint on (challengeId, contributorId) for challengeInterests — use drizzle's `.unique()` or a composite unique index
       - Import `integer`, `date` from drizzle-orm/pg-core as needed

    5. **Matching service** (`packages/server/src/services/matching.service.ts`):
       - `scoreContributorForChallenge(contributorSkills, contributorDomains, challengeSkills, challengeDomain)` -> number (0-100). Skill match = (intersecting skills / challenge skills) * 70, domain match = 30 if domain in contributor domains. Case-insensitive comparison. Return min(100, skillScore + domainScore). If challenge has no skills, skill component is 0.
       - `suggestTeamCompositions(candidates, challengeSkills, circleSize)` -> TeamComposition[]. Generate up to 3 compositions: (a) top-scoring, (b) greedy coverage maximisation for skill diversity, (c) balanced mix (top half score + remainder from diverse). Deduplicate if compositions overlap completely. Score each composition with coverageScore (% required skills covered), diversityScore (unique skills ratio), balanceScore (50% coverage + 30% diversity + 20% avg match).
       - Follow RESEARCH Patterns 4b and the `suggestTeamCompositions` code. Type the candidate parameter correctly: `{ id: string; name: string; skills: string[]; score: number }[]`.

    6. Run `npx drizzle-kit push` from packages/server to push schema to database. If push fails due to enum conflicts, use `npx drizzle-kit generate` then `npx drizzle-kit migrate` instead.

<task type="auto">
  <name>Task 2: Challenge API routes</name>
  <files>
    packages/server/src/routes/challenges.ts
    packages/server/src/express-app.ts
  </files>
  <action>
    Create `packages/server/src/routes/challenges.ts` with an Express Router mounted at `/api/challenges`. Use `authMiddleware` for all routes; use `requireRole("community_manager")` for CM-only routes. Import scoring functions from matching.service.ts.

    **Routes to implement:**

    1. **GET `/`** (feed) — Authenticated. Accepts query params: domain, type, page (default 1), limit (default 20).
       - Fetch contributor's skills and domainPreferences from contributorProfiles.
       - Fetch all open challenges matching WHERE conditions (status = 'open', optional domain/type filters).
       - Score each challenge against contributor using `scoreContributorForChallenge` in TypeScript (not raw SQL — per RESEARCH anti-pattern on JSONB).
       - Sort by score descending, then createdAt descending.
       - Paginate: slice results by page/limit, set hasMore = results.length === limit.
       - Also fetch the current contributor's interest status for each challenge in the page (LEFT JOIN or separate query on challengeInterests where contributorId matches and status = 'active'). Include `myInterest: { status, note } | null` on each challenge in response.
       - Return `{ challenges: [...], page, hasMore }`. Do NOT include matchScore in the response (scores are internal only per user decision).

    2. **GET `/:id`** (single challenge) — Authenticated. Return full challenge object with myInterest status.

    3. **POST `/`** (create) — CM only. Validate body with `createChallengeSchema`. Insert with createdBy = req.contributor.id, status = 'open'. Return 201 with created challenge.

    4. **PUT `/:id`** (update) — CM only. Validate with updateChallengeSchema. **Edit lock**: if challenge.interestCount > 0, reject with 409 and message "Challenge cannot be edited after interest has been expressed. You can close or archive it." Allow only status changes (close/archive) when locked. Return updated challenge.

    5. **POST `/:id/interest`** (toggle interest) — Authenticated.
       - Parse optional `{ note }` from body with interestNoteSchema.
       - Check for existing interest record (unique on challengeId + contributorId).
       - If active: withdraw (set status='withdrawn', lastWithdrawnAt=now(), decrement interestCount with SQL atomic `- 1`). Return `{ status: 'withdrawn' }`.
       - If withdrawn: check 24-hour cooldown from lastWithdrawnAt. If within cooldown, return 429 with cooldownRemainingHours. Otherwise re-activate (set status='active', update note, increment interestCount with SQL atomic `+ 1`).
       - If no existing record: compute matchScore via scoreContributorForChallenge, insert new interest, increment interestCount. Return 201.
       - In success response for active status, include activeInterestCount (count of contributor's active interests across ALL challenges) and maxCircles (from contributor profile) so frontend can show soft capacity warning per RESEARCH Pitfall 6.
       - Return `{ status, activeInterestCount, maxCircles }`.

    6. **GET `/:id/interests`** (list interested contributors) — Authenticated.
       - Fetch all active interests joined with contributors table (for name).
       - Conditional name visibility: CM and admin see all names. Regular contributor sees own name only; others show as null. (Phase 4 will add: also show names of prior Circle members.)
       - Return `{ interests: [{ id, name|null, note|null, isYou, createdAt }], count }`.

    7. **GET `/:id/team-suggestions`** — CM only.
       - Fetch all active interests for the challenge, joined with contributor profiles for skills/name.
       - Score each interested contributor against the challenge.
       - Call `suggestTeamCompositions(candidates, challenge.skillsNeeded, challenge.circleSize)`.
       - Return `{ compositions: TeamComposition[] }`.

    8. **GET `/my-interests`** — Authenticated. Return all challenges the current contributor has active interest in (join challengeInterests with challenges). Useful for the "my interests" view.

    **Mount in express-app.ts:**
    - Import challengeRoutes from `./routes/challenges.js`
    - Add `app.use("/api/challenges", challengeRoutes)` alongside existing routes.

    **Important anti-patterns to avoid (from RESEARCH):**
    - Do NOT expose matchScore in feed response
    - Do NOT use arrayOverlaps on jsonb columns (Drizzle bug #4935) — score in TypeScript
    - Do NOT use read-modify-write for interestCount — use SQL atomic increment/decrement
    - Do NOT block interest expression server-side (soft capacity warning is frontend-only)
