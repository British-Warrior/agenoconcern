  <action>
    **Install dependencies** (from packages/server):
    ```
    npm install @aws-sdk/client-s3 @aws-sdk/s3-request-presigner multer pdf-parse mammoth tesseract.js openai stripe
    npm install -D @types/multer
    ```

    **1. S3 service** (`packages/server/src/services/s3.service.ts`):
    - Import S3Client, PutObjectCommand, GetObjectCommand from @aws-sdk/client-s3
    - Import getSignedUrl from @aws-sdk/s3-request-presigner
    - Lazy S3 client creation (only when env vars present, like existing auth service pattern)
    - `generateUploadUrl(contributorId, fileName, mimeType)` -> { uploadUrl, s3Key } with 5-min expiry. S3 key format: `cvs/{contributorId}/{timestamp}.{ext}`
    - `getObjectBuffer(s3Key)` -> Buffer (downloads file from S3 for text extraction)
    - If S3 env vars missing, functions throw with clear "S3 not configured" message (follow existing graceful degradation pattern from 01-03)

    **2. CV extraction service** (`packages/server/src/services/cv.service.ts`):
    - `extractText(buffer, mimeType)` -> string: dispatches to pdf-parse (application/pdf), mammoth.extractRawText (application/vnd.openxmlformats-officedocument.wordprocessingml.document), or buffer.toString("utf-8") for text/plain. Throws on unsupported type.
    - Wrap pdf-parse in Promise.race with 15-second timeout to prevent hanging on corrupted PDFs (per research pitfall #4)
    - `ocrImage(buffer)` -> string: uses Tesseract.recognize(buffer, "eng"). For server-side use with buffer input (per research pitfall #1).
    - Truncate extracted text to 8000 characters max before returning (per research recommendation, prevents LLM token issues)

    **3. LLM parsing service** (`packages/server/src/services/llm.service.ts`):
    - Import OpenAI and zodResponseFormat from openai/helpers/zod
    - Define `parsedCvSchema` Zod object with: name (string), rolesAndTitles (string[]), skills (string[]), qualifications (string[]), sectors (string[]), yearsOfExperience (number), professionalSummary (string), affirmationMessage (string)
    - `parseCvText(cvText)` -> ParsedCV: calls client.beta.chat.completions.parse with model "gpt-4o-mini", system prompt instructs to extract all fields and generate a warm, specific affirmation message highlighting what makes this person's experience valuable. Use zodResponseFormat for guaranteed schema adherence.
    - If OPENAI_API_KEY is empty, throw "OpenAI not configured" (graceful degradation)

    **4. Stripe Connect service** (`packages/server/src/services/stripe.service.ts`):
    - `createConnectAccount(contributorId, email?)` -> stripeAccountId: creates Express connected account with metadata
    - `createAccountLink(stripeAccountId, refreshUrl, returnUrl)` -> url: generates single-use account link (NEVER cache -- per research pitfall #2)
    - `getAccountStatus(stripeAccountId)` -> { detailsSubmitted, chargesEnabled, payoutsEnabled }
    - If STRIPE_SECRET_KEY is empty, throw "Stripe not configured" (graceful degradation)

    **5. Onboarding routes** (`packages/server/src/routes/onboarding.ts`):
    Create Express router with these endpoints (all behind authMiddleware):

    - `POST /upload-url`: accepts { fileName, mimeType }, validates mimeType against allowlist (PDF, DOCX, TXT only -- images go through upload-image), returns { uploadUrl, s3Key }
    - `POST /upload-image`: uses multer(memoryStorage, 10MB limit).single("file"), accepts JPG/PNG, runs ocrImage on buffer, uploads original to S3 for record, returns { s3Key, extractedText }
    - `POST /start-parse`: accepts { s3Key, mimeType } or { extractedText } (for image path). Creates cvParseJobs row with "processing" status. Kicks off async processing (NOT awaited in request handler -- per research anti-pattern): downloads from S3 -> extractText -> parseCvText -> upsert contributorProfiles -> update job status to "complete". On error, sets job status to "failed" with errorMessage. Returns { jobId }.
    - `GET /parse-status/:jobId`: returns job { id, status, errorMessage }. When status is "complete", also returns the parsed profile data from contributorProfiles.
    - `GET /profile`: returns the current contributor's profile from contributorProfiles (or 404 if not found)
    - `PUT /profile`: accepts updateProfileSchema body, updates contributorProfiles fields. Returns updated profile.
    - `PUT /preferences`: accepts preferencesSchema body, updates contributorProfiles preference fields + sets contributor status to "active" in contributors table (this is the onboarding completion transition -- per research pitfall #6). Returns updated profile.
    - `POST /stripe/connect`: creates Stripe Connect Express account, saves stripeAccountId to profile, generates account link with refresh/return URLs, returns { url }
    - `GET /stripe/return`: retrieves account status from Stripe, updates stripeStatus in profile ("complete" if detailsSubmitted && chargesEnabled, else "pending"), redirects to CLIENT_URL/onboarding/complete
    - `POST /stripe/skip`: sets stripeStatus to "not_started" (no-op if already), sets contributor status to "active", returns 200

    **6. Mount routes** in express-app.ts:
    - Import onboardingRoutes, mount at `/api/onboarding`
    - Add `app.use(express.urlencoded({ extended: true }))` if not present (needed for multer)

    Follow existing patterns: use getDb() for database access, getEnv() for config, same error handling style as auth routes.
