## v1.2 Requirements

Requirements for v1.2 Institution Management & iThink Integration. Each maps to roadmap phases.

### Institution Management

- [x] **INST-01**: CM can view a list of all institutions with name, city, and active status
- [x] **INST-02**: CM can create a new institution with name, description, and city
- [x] **INST-03**: CM can edit an existing institution's name, description, and city
- [x] **INST-04**: CM can activate or deactivate an institution
- [x] **INST-05**: CM can assign a contributor to an institution
- [x] **INST-06**: CM can remove a contributor's institution assignment
- [x] **INST-07**: CM can view all contributors assigned to a specific institution
- [x] **INST-08**: Institution landing page displays live aggregated stats (contributor count, challenges, hours) instead of static JSONB

### Webhook Integration

- [x] **WHOOK-01**: IU exposes a POST /api/webhooks/ithink endpoint that accepts signed payloads from iThink
- [x] **WHOOK-02**: Webhook receiver verifies HMAC-SHA256 signature using timingSafeEqual
- [x] **WHOOK-03**: Webhook receiver rejects requests with timestamps older than 5 minutes (replay protection)
- [x] **WHOOK-04**: Webhook receiver deduplicates deliveries via webhook_deliveries table (idempotent)
- [x] **WHOOK-05**: Webhook receiver validates payload schema with Zod before processing
- [x] **WHOOK-06**: Webhook receiver verifies contributor-institution relationship before writing any flag
- [x] **WHOOK-07**: iThink dispatches signed webhooks when a screening flags a contributor as needing attention
- [x] **WHOOK-08**: Admin can rotate the webhook shared secret without downtime (dual-secret transition window)

### Attention

- [x] **ATTN-01**: Webhook receiver stores per-contributor attention flags with audit fields (cleared_by, follow_up_notes)
- [x] **ATTN-02**: CM can view a list of flagged contributors filtered by their institution
- [x] **ATTN-03**: CM can clear an attention flag with follow-up notes recorded
- [x] **ATTN-04**: CM can view attention signal history per institution ordered by date

### Reporting

- [x] **PDF-01**: CM can generate a PDF impact report for an institution on demand
- [x] **PDF-02**: PDF report includes institution name, date range, contributor count, challenge count, and total hours
- [x] **PDF-03**: PDF report is streamed to the browser (not stored in S3)
- [x] **PDF-04**: PDF report is styled with platform branding

## Future Requirements (carried forward)

### v1.2.x
- **WELL-01**: PDF impact report includes anonymous wellbeing aggregate band (min 5 contributors threshold)
- **ATTN-05**: CM can view attention signal trend (frequency increasing/decreasing) per institution

### v2+
- **PRIV-01**: Contributor-level wellbeing data visible to CM (requires GDPR impact assessment)
- **PDF-05**: PDF report scheduling and auto-delivery to institution contact email
- **PORTAL-01**: Institution portal login (institutions access own data without CM mediation)
