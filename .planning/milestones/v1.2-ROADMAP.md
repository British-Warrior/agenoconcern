# Milestone v1.2: Institution Management & iThink Integration

**Status:** SHIPPED 2026-03-24
**Phases:** 12-15
**Total Plans:** 10

## Overview

Give the CM operational tools to manage institution-contributor relationships with live impact data and PDF reports, and integrate iThink so contributors flagged as needing attention surface in a CM attention dashboard.

## Phases

### Phase 12: Institution Data Foundation

**Goal:** The CM can manage institution records and contributor assignments, and institution landing pages display live aggregated stats derived from real contributor data.
**Depends on:** Phase 11 (institutions table and statsJson JSONB exist)
**Plans:** 3 plans

- [x] 12-01: Junction table migration (contributor_institutions many-to-many), Drizzle schema, shared Zod schemas
- [x] 12-02: Institution CRUD API endpoints, CM route guard, and CM institution management page with inline editing
- [x] 12-03: Contributor-institution assignment endpoints, CM assignment UI, live stats aggregation, and landing page update

### Phase 13: iThink Webhook Integration

**Goal:** iThink can dispatch signed webhooks when a screening flags a contributor as needing attention, and IU receives, validates, and stores those signals with a complete security stack in place before any flag is written to the database.
**Depends on:** Phase 12 (institution_id FK must exist on contributors)
**Plans:** 3 plans

- [x] 13-01: DB schema (ithink_attention_flags table, webhook_deliveries idempotency table) and ITHINK_WEBHOOK_SECRET env var
- [x] 13-02: IU webhook receiver (POST /api/webhooks/ithink) with full security stack (raw body, timingSafeEqual, timestamp window, idempotency, Zod validation, relationship check, flag insertion)
- [x] 13-03: iThink-side webhook dispatch (HMAC signing, fire-and-forget, exponential backoff retry) and secret rotation

### Phase 14: CM Attention Dashboard

**Goal:** The CM can see which contributors at their institution have been flagged by iThink, view the signal history, and clear flags with follow-up notes recorded — all scoped strictly to the CM's own institution.
**Depends on:** Phase 13 (ithink_attention_flags table must exist)
**Plans:** 2 plans

- [x] 14-01: Attention API routes (GET /api/attention and POST /api/attention/:flagId/resolve) with institution scope enforced from DB
- [x] 14-02: CM attention dashboard page (AttentionDashboard.tsx), TanStack Query hooks, confirm-before-resolve action, and signal history view

### Phase 15: PDF Impact Report

**Goal:** The CM can generate a branded PDF impact report for any institution on demand, and the report is streamed directly to the browser without being stored anywhere.
**Depends on:** Phase 12 (institution-contributor data and live aggregation queries)
**Plans:** 2 plans

- [x] 15-01: pdfkit setup, Inter fonts, institution-report.ts PDF builder, GET /api/admin/institutions/:slug/report.pdf streaming route
- [x] 15-02: CM UI — "Generate Report" button with loading state, date range inputs, raw-fetch blob download

## Milestone Summary

**Key Decisions:**
- Many-to-many junction table for contributor-institution (not single FK) — enables future multi-institution assignments
- CMRoute guard pattern: check role community_manager OR admin, redirect to /dashboard
- adminRouter uses router.use(authMiddleware, requireRole) at top — all routes automatically protected
- Live stats return null when no contributors assigned — frontend hides stats section entirely
- pdfkit ^0.18.0 chosen over @react-pdf/renderer (ESM breakage) and Puppeteer (300MB binary)
- HMAC webhook verification: sign over timestamp.rawBody, timestamp check AFTER signature verification
- ITHINK_WEBHOOK_SECRET has no .default() — server refuses to start without it
- Dual-secret rotation via optional ITHINK_WEBHOOK_SECRET_PREV
- Attention routes: /history registered before /:flagId to prevent Express param collision
- Inter TTFs sourced from rsms/inter v4.1 GitHub release (not @fontsource which ships WOFF only)
- Manual text layout for PDF stats — simpler than pdfkit's undocumented table API
- Raw fetch + blob for binary PDF download — apiClient only handles JSON

**Issues Resolved:**
- None — clean execution across all 4 phases

**Issues Deferred:**
- Employment Agencies Act 1973 classification needs legal advice before payments go live (carried from v1.1)

**Technical Debt Incurred:**
- LIMIT 1 on CM institution lookup is a pilot-scale assumption (one CM manages one institution)
- statsJson preserved in DB as cache/fallback but never returned to client — could be removed in future cleanup
