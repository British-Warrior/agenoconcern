# Milestone v1.1: Pilot-Ready

**Status:** SHIPPED 2026-03-21
**Phases:** 7-11
**Total Plans:** 11

## Overview

Make the platform usable for real pilot deployment — fix critical UX issues, enable VANTAGE AI overlay, add kiosk mode for institutional embedding, and give challengers their own portal.

## Phases

### Phase 7: UX Fixes

**Goal:** The platform is navigable, readable, and credible for institutional demos — no stranded users, no empty screens, no broken affordances.
**Depends on:** Nothing
**Plans:** 3 plans

- [x] 07-01: Navigation links + route constants + hover affordances
- [x] 07-02: Dashboard summary cards + wellbeing score interpretation
- [x] 07-03: CM role-conditional circles + error sanitisation + isolated bugs

### Phase 8: Wellbeing Visualisation

**Goal:** Wellbeing data is interpretable and funder-ready — contributors see trend, context, and UK benchmarks rather than raw numbers.
**Depends on:** Phase 7
**Plans:** 1 plan

- [x] 08-01: Install recharts, build WellbeingChart component, integrate into ImpactDashboard

### Phase 9: Server Foundation and VANTAGE

**Goal:** VANTAGE can call the platform's REST API with a secure API key, and the codebase has the type and schema foundation needed to build the challenger portal.
**Depends on:** Phase 7
**Plans:** 2 plans

- [x] 09-01: Schema changes (challenger enum, api_keys table, challenger_organisations table) + Drizzle migration
- [x] 09-02: API key middleware + rate limiter + VANTAGE routes + contract document

### Phase 10: Challenger Portal

**Goal:** Organisations can self-service submit challenges, track their progress, and rate completed resolutions — without requiring CM manual intake.
**Depends on:** Phase 9
**Plans:** 2 plans

- [x] 10-01: Schema migration + shared types + challenger server routes
- [x] 10-02: React challenger portal pages, hooks, API client, and routing

### Phase 11: Kiosk Mode and Institutional Pages

**Goal:** The platform is safely embeddable in libraries and community centres — shared computers auto-clean sessions, and each institution has a public entry point.
**Depends on:** Phase 9, Phase 7
**Plans:** 3 plans

- [x] 11-01: Kiosk mode: KioskContext + idle timer + UI modifications + End Session
- [x] 11-02: Institution backend: schema + migration + public API endpoint
- [x] 11-03: Institution landing page + route wiring + kiosk auto-activation

## Milestone Summary

**Key Decisions:**

- Module-level kiosk flag persistence (survives SPA navigation without threading ?kiosk=true through every navigate())
- MVP institution stats as JSONB (no contributor-institution FK, seed stats for pilot)
- react-idle-timer over custom event listeners (handles edge cases)
- Reuse existing POST /api/auth/logout for kiosk (client-side difference is queryClient.clear())
- ChallengerRoute redirects unauthenticated to /challenger/register (not /login)
- Manual SQL migrations (Drizzle journal empty while DB had existing tables)

**Issues Resolved:**

- 11 UX issues from walkthrough fixed (navigation, dashboard, CM views, affordances, errors)
- TypeScript type errors in challenger portal (instanceof check, response envelope unwrapping)
- Auto-create challenger org when switching via dev-role

**Issues Deferred:**

- Live institution stats aggregation (needs contributor-institution FK — v2)
- express-rate-limit IPv6 validation warning (cosmetic)
- ROUTES.INSTITUTION constants unused (for future admin panel)

**Technical Debt Incurred:**

- Phase 10 missing formal VERIFICATION.md (human checkpoint was sufficient)
- Institution stats are seeded JSONB, not live aggregated
- express-rate-limit IPv6 warning on startup
